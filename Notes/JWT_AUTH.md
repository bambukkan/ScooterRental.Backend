# Система проката самокатов (Rental System)

## Шпаргалка по JWT и Авторизации

1. **Регситрация:** Происходит через обычный метод add. Из тела приходит dto, которое имеет имя, фамилию пароль и имейл. 
Происходит хэширование пароля, благодаря классу PasswordHasher. Записываем захэшированный пароль прямо в пароль юзера, чтобы это на бд отправилось. Ведь пароль никто не может знать, кроме как пользователь
2. **Логин:** Получаем токен из сервиса. Чтобы в сервисе сделать токен, мы получаем пользователя по имейлу. Делаем проверки на то, что существует пользователь и совпадают ли пароли. Если все в порядке, создаем токен благодаря JwtProvider классу, который генерирует токен.
3. **JwtProvider** Этот класс имеет поле, который имеет секретный ключ и раз в сколько времени обновлять токен. Получать мы их будем делая конфигурацию в програм.cs, благодаря тому что в appsettings.json записываем их заранее. 
Записываем полезную информацию в claims в виде ключ значение, где в значение можно записать полезную инфу о пользователе, но только ни в коем случае пароль! Ведь эта полезная инфа в открытом доступе
Короче, делаем секретный ключ в байты. С помощью new SymmetricSecurityKey делаем симметричный ключ безопасности , который будет такой же как и для токена, так и для проверки этого самого токена в будущем
Потом связываем этот серетный ключ и его шифрование в одну переменную
И вот после этих махинаций мы создаем токен по трем правилам: 
1. Полезная информация
2. Сколько токен живет
3. Цифровая подпись

4. **Применение:** Чтобы все что мы написали применить в программе мы используем расширение которое будет делать правила для проверки токена, чтобы можно было авторизоваться. Так же когда мы логинимся и пытаемся вызвать методы других контроллеров без options.Events, то пишет что мы не авторизованы. Чтение токена (options.Events): По умолчанию .NET ищет токен только в HTTP-заголовках. Мы добавляем событие OnMessageReceived, чтобы при каждом новом запросе к закрытым методам (например, самокатам) .NET автоматически заглядывал в куки "tasty-cookies", доставал оттуда токен и проверял его

# Написанное от нейронки, исходя из моего

1. **Регистрация:** Происходит через метод `Add`. Из тела запроса приходит DTO (имя, фамилия, пароль, email). Пароль хэшируется через класс `PasswordHasher`. Записываем полученный хэш в поле `PasswordHash` сущности пользователя и сохраняем в БД. Чистый пароль никто, кроме пользователя, не знает.

2. **Логин:** Пользователь присылает Email и Пароль. Сервис находит юзера по email, проверяет существование и через `PasswordHasher.Verify` сравнивает пароли. Если всё ок, вызывается `JwtProvider` для генерации токена.

3. **JwtProvider (Создание токена):** * Получает `SecretKey` и время жизни токена из `appsettings.json` через `IOptions`.
   * **Claims (Payload):** Открытая полезная информация (например, `UserID`). Пароли сюда класть категорически нельзя!
   * **Ключ:** Текстовый `SecretKey` переводится в байты. С помощью `new SymmetricSecurityKey` создается симметричный ключ (один и тот же для создания и для проверки).
   * Токен собирается по 3 составляющим: 
     1. Claims (Полезная информация)
     2. Expires (Время жизни токена)
     3. SigningCredentials (Цифровая подпись: ключ + алгоритм HMAC-SHA256 для защиты от подделки).
   * С помощью `JwtSecurityTokenHandler().WriteToken(token)` объект превращается в компактную Base64-строку.

4. **Применение и чтение из Куки:**
   * Контроллер при логине складывает токен в куки: `Response.Cookies.Append("tasty-cookies", token, new CookieOptions { HttpOnly = true })`. Флаг `HttpOnly` защищает токен от кражи скриптами.
   * На закрытые контроллеры (Самокаты, Аренда) вешается атрибут `[Authorize]`.
   * В расширении `ApiExtensions` мы настраиваем правила проверки (`TokenValidationParameters`). 
   * **Важно:** Чтобы .NET умел читать токен из куки, мы настраиваем `options.Events (OnMessageReceived)`. Этот перехватчик при каждом запросе достает токен из куки `"tasty-cookies"` и передает его встроенной системе проверки .NET.
