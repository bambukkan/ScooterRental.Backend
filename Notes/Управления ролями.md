## 💎 Главная фишка: Почему больше НЕ НАДО передавать `UserId` в запросах?

Раньше, чтобы создать аренду (Booking), фронтенд или Swagger должны были явно присылать JSON вида:
```json
{
  "scooterId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "userId": "7ba94f11-1111-2222-3333-4c963f66afb2"  // <-- СТАРАЯ ДЫРА В БЕЗОПАСНОСТИ
}

```

### В чем была проблема?

Это небезопасно. Любой пользователь (или хакер через Postman) мог подставить в поле `userId` чужой Guid. Сервер бы поверил на слово и повесил аренду (или списание денег) на случайного человека.

### Как это работает теперь через JWT (Магия Claims)

Когда пользователь успешно залогинился (`POST Users/login`), класс `JwtProvider` зашивает его уникальный ID внутрь самого токена в виде так называемого **Claim** (заявления о личности):

```csharp
Claim[] claims = [ new("UserID", user.Id.ToString()) ];

```

Этот токен подписывается секретным ключом сервера (Signature), кодируется и отправляется в браузер в куке `tasty-cookies`. Подделать токен невозможно — при изменении хотя бы одного символа подпись сломается, и сервер его отвергнет.

Теперь, когда пользователь делает запрос (например, «Арендовать самокат»), кука автоматически прикрепляется к запросу.

На бэкенде происходит следующее:

1. Мидлварь `UseAuthentication()` перехватывает запрос, достает токен из куки `tasty-cookies` и проверяет его валидность.
2. Если токен «живой», .NET расшифровывает блок данных (Payload) и автоматически заполняет встроенное свойство **`User`**, которое есть в каждом контроллере.
3. Из этого свойства `User` мы можем в любой момент достать `UserID` прямо в коде эндпоинта.

## 🛠 Реализация в коде (Шпаргалка)

### 1. Чистый DTO (`CreatingBookingRequest`)

Из тела запроса полностью убирается упоминание юзера. Фронтенд присылает только то, что касается самого действия:

```csharp
public record CreatingBookingRequest(Guid ScooterId);

```

### 2. Извлечение данных в Контроллере (`BookingController`)

Мы вешаем атрибут `[Authorize]`. Если токена нет — метод даже не начнет выполняться (вернется `401 Unauthorized`). Если токен есть — вытаскиваем зашитый ID:

```csharp
[HttpPost]
[Authorize] // Доступ только для вошедших
public async Task<IActionResult> CreateBooking([FromBody] CreatingBookingRequest request)
{
    // Достаем строковое значение клейма "UserID", которое зашил JwtProvider
    var userIdClaim = User.FindFirst("UserID")?.Value;
    
    if (userIdClaim == null)
    {
        return Unauthorized();
    }

    // Парсим строку обратно в Guid
    Guid userId = Guid.Parse(userIdClaim);

    // Передаем в сервис уже ОЧИЩЕННЫЙ, проверенный системой userId
    var bookingId = await _bookingService.Add(userId, request); 
    
    return Ok(bookingId);
}

```

### 3. Обработка в Сервисе (`BookingService`)

Метод сервиса теперь принимает два параметра отдельно: чистый `Guid userId` из токена и данные запроса.

```csharp
public async Task<Guid> Add(Guid userId, CreatingBookingRequest request)
{
    var booking = new BookingEntity
    {
        Id = Guid.NewGuid(),
        UserId = userId, // Железно защищено, подмена невозможна
        ScooterId = request.ScooterId,
        StartTime = DateTime.UtcNow
    };

    await _bookingRepository.Add(booking);
    return booking.Id;
}

```

## 🧠 Что это дало архитектуре?

1. **Защита от подмены (ID Spoofing):** Пользователь может совершать действия (аренда, обновление профиля, удаление) **только от своего имени**. Он физически не может передать чужой ID, потому что сервер берет ID только из подписанного паспорта-токена.
2. **Красивый и лаконичный API:** Меньше мусора в JSON-запросах.
3. **Удобство фронтенда:** Разработчику фронтенда не нужно вручную вытягивать ID текущего юзера из памяти и подставлять его в каждый эндпоинт. Браузер сам заботится о куках.
